## The problem

LLM applications fail silently. A RAG pipeline returns confidently wrong answers.
A support agent skips a required tool. A prompt has weak approval rules. A schema
response looks valid until a downstream service rejects it. Observability can
show that something happened; DebugAI is built to say what failed, where it
failed, and what to change.

## What DebugAI does

DebugAI is an **LLM failure diagnosis engine**. Given a prompt, output, and
optional runtime context, it computes deterministic signals, runs detector
layers, and returns a ranked diagnosis with evidence and a fix.

```python
from debugai import debug_report

report = debug_report(
    prompt="What is the refund policy for electronics?",
    output="Electronics can be returned within 90 days for a full cash refund.",
    chunks=["Store hours are 9am to 5pm.", "Parking is behind the building."],
    similarity_scores=[0.42, 0.40],
    run_fix=True,
)

print(report["failure"])        # retrieval_failure
print(report["evidence"])       # short evidence bullets
print(report["fix"])            # specific repair guidance
print(report["regression_artifact"])
```

<Note>
  Detection is deterministic by default: no LLM, no randomness, and stable
  output for the same input. Optional LLM layers are available for explanations,
  instruction judging, and dynamic prompt-audit probes.
</Note>

## Failure coverage

DebugAI starts from 8 core signals: context-output overlap, entity coverage,
retrieval similarity, contradiction, output variance, latency, token-usage
ratio, and context-length ratio. It also computes auxiliary pipeline and
security fields such as retrieval margin, query drift, claim support, context
dilution, source conflict, tool-argument risk, instruction conflict, and refusal
signals.

The detector layer covers:

| Layer | Failures |
|---|---|
| Retrieval | `retrieval_failure`, `retrieval_ambiguity`, `query_drift` |
| Grounding | `hallucination`, `entity_gap`, `citation_failure` |
| Contracts | `schema_violation`, `tool_call_failure`, `tool_result_ignored` |
| Runtime | `context_overflow` |
| Prompt behavior | `prompt_brittleness`, `ambiguous_prompt`, `instruction_violation` |
| Safety | `prompt_injection`, `sensitive_data_leak` |

## Debugging layers

```
  Capture (prompt, output, chunks, tools, schema, runtime)
        │
        ▼
  ┌──────────────────────┐
  │  Signals              │ deterministic metrics + pipeline fields
  └─────────┬────────────┘
            │
        ▼
  ┌──────────────────────┐
  │  Detectors            │ retrieval, grounding, schema, tools, safety
  └─────────┬────────────┘
            │
        ▼
  ┌──────────────────────┐
  │  Fix artifact         │ root cause, fix, regression test skeleton
  └─────────┬────────────┘
            │
        ▼
  ┌──────────────────────┐
  │  Optional LLM layers  │ explainer, judge, dynamic prompt audit
  └──────────────────────┘
```

## Beyond single-call diagnosis

- `audit_prompt()` finds static prompt vulnerabilities and can run dynamic attack
  probes against your own harness.
- `analyze_pipeline()` points to the failing stage in a query rewrite,
  retrieval, tool, generation, or validation trace.
- `evaluate_corpus_file()` scores a labeled failure corpus so you can track
  detector accuracy over time.
- `debug_report()` returns the product-level artifact most teams should log in
  CI, support tickets, and incident reviews.
