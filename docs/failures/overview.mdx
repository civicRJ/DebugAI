---
title: Failure Types
description: The detector layers DebugAI runs and what each failure means.
---

DebugAI's rule engine runs all detectors on every request, ranks them by
confidence, and returns a primary failure plus secondary issues. Detection is
deterministic unless you explicitly enable optional LLM layers such as the
instruction judge.

## Detector coverage

| Failure | Layer | Severity | What it catches |
|---|---|---|---|
| `context_overflow` | runtime | Critical | Prompt/context fills the model window and likely truncates content |
| `sensitive_data_leak` | safety | Critical | Output exposes API keys, tokens, credentials, PII, or secret-like values |
| `schema_violation` | schema | Critical | Output violates the required JSON Schema |
| `tool_call_failure` | tool execution | Critical | Required tool missing, wrong tool used, malformed args, or tool error |
| `tool_result_ignored` | tool execution | Critical | Tool returned usable evidence but final answer ignored it |
| `prompt_injection` | safety | Critical | Retrieved context contains instruction-like jailbreak text |
| `query_drift` | retrieval | Warning | Query rewrite drifted away from the user request |
| `retrieval_failure` | retrieval | Critical | Retrieved chunks do not match the query |
| `retrieval_ambiguity` | retrieval | Warning | Top retrieved chunks are too close to rank confidently |
| `citation_failure` | citation | Warning | Missing or invalid citations against retrieved chunks |
| `entity_gap` | knowledge base | Warning | Output mentions entities absent from retrieved context |
| `hallucination` | grounding | Critical | Retrieval succeeded but output is unsupported or contradictory |
| `prompt_brittleness` | prompt | Warning | Grounding is healthy but outputs are unstable |
| `ambiguous_prompt` | prompt | Warning | User request is underspecified but the model answered instead of clarifying |
| `instruction_violation` | prompt behavior | Critical | Optional judge finds the output broke system-prompt rules |

## Confidence scores

Each detector reports a confidence from 0 to 1. Scores are based on which gates
fired and how strongly the related signals support the failure.

- `0.90+`: strong gated failure with multiple supporting signals
- `0.70-0.89`: clear failure with one or more supporting signals
- `0.60-0.69`: warning or borderline case worth reviewing

## Reading a result

```python
from debugai import analyze

result = analyze(prompt=..., output=..., chunks=..., similarity_scores=...)

if not result["healthy"]:
    p = result["primary"]
    print(p["failure"])     # e.g. "retrieval_failure"
    print(p["layer"])       # e.g. "retrieval"
    print(p["confidence"])  # e.g. 0.95
    print(p["severity"])    # "critical" or "warning"
    print(p["fix"])
    print(p["evidence"])

    for s in result["secondary"]:
        print(s["failure"], s["confidence"])
```

## Stage-level failures

Use [Pipeline traces](/sdk/pipeline) when you need to map a failure back to the
stage that produced it. A final `hallucination` can originate from retrieval,
query rewrite, generation, validation, or tool execution; the pipeline analyzer
keeps those causes separate.
