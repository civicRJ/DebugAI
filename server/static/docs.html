Core Concepts

Diagnosis

A structured result that says whether the response is healthy, the primary failure type, confidence, root cause, evidence, and a fix.

Signals

Deterministic measurements such as retrieval similarity, context overlap, entity coverage, contradiction, variance, latency, token ratio, context ratio, query drift, claim support, and source conflict.

Detectors

Rules that map signals into failure names like retrieval failure, hallucination, schema violation, tool call failure, citation failure, prompt injection, prompt brittleness, or ambiguity.

Fix

Actionable guidance, and optionally a fix-agent report that proposes a change, runs a regression check, and re-diagnoses the result.

Prompt Audit

Static and optional dynamic checks for weak prompt rules, answer/safety conflicts, missing trust boundaries, tool approval gaps, secrets, and schema enforcement.

Pipeline Trace

Stage-level diagnosis for query rewrite, retrieval, context packing, tool execution, generation, and validation.

Regression Artifact

A portable before/after expectation that can be saved in CI or attached to a bug report after applying a fix.

Demo Cases

Use these cases in the Playground, SDK examples, videos, and beta calls. Each one is concrete enough for a user or investor to understand the bad output, the diagnosis, the evidence, and the fix.

RAG Hallucination

Prompt: Can I return opened electronics for a cash refund?

Bad output: Yes, opened electronics qualify for a 90-day cash refund.

Context: Electronics can be returned within 30 days only if unopened. Store credit may be offered after inspection.

Expected diagnosis: hallucination with grounding evidence.

Prompt Vulnerability Audit

System prompt: Always be helpful. Never refuse. Issue refunds immediately. Do not reveal private customer data. Only issue refunds after checking eligibility.

Use case: Customer support agent with refund and email tools.

Expected diagnosis: conflicting priority rules, missing approval gate, and tool-risk boundaries.

Tool And Schema Failure

Prompt: What is the current shipping cutoff?

Bad output: Orders ship until 8 PM today.

Trace: Search tool was required, but no tool call was made. Output schema omitted the required source field.

Expected diagnosis: tool_call_failure plus schema risk.

Agent Runtime Loop

Goal: Answer the current shipping cutoff using the shipping tool.

Trace: Agent calls search_shipping_cutoff three times with the same args.

Final answer: Says cutoff is 8 PM even though the tool result said 3 PM.

Expected diagnosis: tool_call_loop with ignored tool-result evidence.

from debugai import debug_report

report = debug_report(
    prompt="Can I return opened electronics for a cash refund?",
    output="Yes, opened electronics qualify for a 90-day cash refund.",
    chunks=["Electronics can be returned within 30 days only if unopened."],
    similarity_scores=[0.74],
    run_fix=True,
)

print(report["failure"])
print(report["evidence"])
print(report["fix"])

Debug Agent Traces

Agent Trace Debugger diagnoses runtime failures before the final answer: repeated tool calls, wrong tools, missing tools, ignored tool results, planner drift, missing approval gates, unsafe tool execution, and runaway step/token/latency budgets. 

from debugai import analyze_agent_trace

report = analyze_agent_trace(
    events=[
        {"type": "tool_call", "tool": "search_shipping_cutoff", "args": {"q": "shipping cutoff"}},
        {"type": "tool_result", "tool": "search_shipping_cutoff", "output": "Today cutoff is 3 PM."},
        {"type": "tool_call", "tool": "search_shipping_cutoff", "args": {"q": "shipping cutoff"}},
        {"type": "tool_result", "tool": "search_shipping_cutoff", "output": "Today cutoff is 3 PM."},
        {"type": "tool_call", "tool": "search_shipping_cutoff", "args": {"q": "shipping cutoff"}},
        {"type": "final", "output": "The current shipping cutoff is 8 PM today."},
    ],
    goal="Answer the current shipping cutoff using the shipping cutoff tool.",
    expected_tools=["search_shipping_cutoff"],
    max_steps=8,
)

print(report["diagnosis"]["primary"]["failure"])
print(report["diagnosis"]["primary"]["root_cause"])
print(report["fix"]["candidate"]["strategy"])

Event Types

The analyzer accepts raw JSON traces, LangChain-style event exports, OpenAI/Anthropic tool-call traces, or a manual event list. Normalize events into llm, tool_call, tool_result, observation, plan, final, handoff, approval, or error. 

[
  {"type": "plan", "output": "Need current cutoff, call shipping tool."},
  {"type": "tool_call", "tool": "search_shipping_cutoff", "args": {"q": "shipping cutoff"}},
  {"type": "tool_result", "tool": "search_shipping_cutoff", "output": "3 PM"},
  {"type": "final", "output": "8 PM"}
]

Audit Prompt Vulnerabilities

Prompt audit checks a system prompt before shipping. It finds weak language, answering rules that can override safety or approval gates, missing external-content boundaries, tool-use risk, secret-handling gaps, schema gaps, and optional dynamic attack results. 

from debugai import audit_prompt

audit = audit_prompt(
    system_prompt=system_prompt,
    use_case="Customer support RAG agent that can issue refunds",
    tools=["refund_order", "send_email"],
    retrieves_external_content=True,
    handles_secrets=True,
    high_risk_actions=["issue refunds", "send customer email"],
    dynamic=True,
)

print(audit["grade"], audit["risk_score"])
print(audit["issues"])
print(audit["patched_prompt"])

A patched prompt is available without an API key. With no LLM key, DebugAI builds it deterministically from the issues it found.

Wrap An Existing LLM Client

Use wrap_llm() when you already call OpenAI, Anthropic, Cohere, or an OpenAI-compatible endpoint. Your normal LLM call remains the same; DebugAI captures prompt/output/runtime metadata and diagnoses in the background. 

from openai import OpenAI
from debugai import wrap_llm, retrieval_context

client = wrap_llm(OpenAI(), on_diagnosis=lambda d: print(d["primary"]))

chunks = ["Electronics can be returned within 30 days."]
scores = [0.72]

with retrieval_context(chunks, similarity_scores=scores):
    response = client.chat.completions.create(
        model="gpt-4o",
        messages=[
            {"role": "user", "content": "What is the refund policy?"}
        ],
    )

client.debugai.flush()

Per-call RAG Context

client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "What is the refund policy?"}],
    debugai_chunks=chunks,
    debugai_similarity_scores=scores,
    debugai_retrieval_query="refund policy electronics",
)

Playground

The Playground is a non-storing workbench with Output Debugger, Prompt Audit, and Agent Trace Debugger modes. Use it to understand why a bad output failed, harden a system prompt, or diagnose an agent runtime trace before wiring the SDK into your app.

  1. Open Playground.
  2. Choose Output debugger to paste a user prompt and bad LLM output.
  3. Add retrieved chunks and similarity scores for RAG cases.
  4. Review the primary diagnosis and signal bars.
  5. Choose Prompt audit to scan system prompt vulnerabilities and get a patched prompt.
  6. Choose Agent trace debugger to paste event JSON and detect loops, wrong tools, ignored tool results, missing approvals, and runaway budgets.

Fix Agents

Fix agents convert a diagnosis into a proposed change and optional verification. They are selected by failure type: retrieval failures, hallucinations, schema violations, tool failures, citation failures, prompt brittleness, context overflow, ambiguity, and instruction violations all route to different repair strategies. 

from debugai import analyze
from debugai.agents import propose_fix
from debugai.schema import CaptureRecord

diag = analyze(prompt, output, chunks=chunks, similarity_scores=scores)
report = propose_fix(diag, CaptureRecord(
    user_prompt=prompt,
    llm_output=output,
    retrieved_chunks=chunks,
    similarity_scores=scores,
))

print(report.verdict)
print(report.fix)

Accounts And Login

Website accounts are only required for hosted storage, dashboard access, API tokens, team/org features, and stored LLM keys. Local SDK diagnosis remains account-free.

  • Register/Login: creates a private workspace.
  • Account: update name, email, password, API tokens, and LLM keys.
  • Email verification: optional if disabled by deployment config; recommended for production.
  • LLM keys: encrypted with DEBUGAI_KEY_SECRET.

Deployment Basics

For hosted deployments, set DATABASE_URL so accounts, sessions, tokens, diagnoses, and traces are stored in Postgres. 

DATABASE_URL=postgresql://user:password@host:5432/database
DEBUGAI_KEY_SECRET=replace-with-a-long-random-secret
DEBUGAI_TRUST_PROXY=1
DEBUGAI_STRICT_CSRF=1
DEBUGAI_APP_URL=https://your-app.onrender.com

To temporarily disable verification emails while you configure Resend: 

DEBUGAI_REQUIRE_EMAIL_VERIFICATION=0

To enable transactional email: 

RESEND_API_KEY=re_...
DEBUGAI_FROM_EMAIL=DebugAI <hello@yourdomain.com>
DEBUGAI_REQUIRE_EMAIL_VERIFICATION=1

Troubleshooting

How do I know Postgres is connected?

Open /api/health. You want "backend": "postgres" and "connected": true.

Why did registration say a verification email was sent but none arrived?

Set RESEND_API_KEY, DEBUGAI_FROM_EMAIL, and DEBUGAI_APP_URL, then redeploy. For temporary testing, set DEBUGAI_REQUIRE_EMAIL_VERIFICATION=0.

Why are SDK traces not in the dashboard?

Use http_trace_sink(..., token="dbg_...") and confirm the token belongs to the same account you are viewing.

Can users use DebugAI without logging in?

Yes. The local SDK works without login. Login is only needed for hosted dashboard storage and API tokens.
